# TTL 机制防止路由死循环

## 背景

在 P2P 网络的路由转发过程中，如果出现路由环路（A → B → C → A），消息会无限循环转发，导致网络拥塞和资源浪费。

## 解决方案

实现 TTL（Time To Live）机制，类似于 IP 协议中的 TTL 字段：

1. **初始值**：消息创建时设置 TTL = 128
2. **递减规则**：每经过一跳（每个节点转发），TTL 减 1
3. **过期检查**：TTL ≤ 0 时丢弃消息，返回 502 错误
4. **默认值**：使用常量 `DEFAULT_TTL = 128`

## 实现要点

### 1. 类型定义
在 `RelayMessage` 接口中添加可选字段：
```typescript
export interface RelayMessage {
  // ... 其他字段
  ttl?: number;
}
```

### 2. 发送消息时设置 TTL
所有创建 `RelayMessage` 的地方都设置初始 TTL：
```typescript
const message: RelayMessage = {
  type: 'relay-request',
  // ... 其他字段
  ttl: DEFAULT_TTL,
};
```

### 3. 转发时递减 TTL
在转发消息前递减 TTL 并检查：
```typescript
const currentTTL = message.ttl ?? DEFAULT_TTL;
if (currentTTL <= 0) {
  // TTL 过期，丢弃消息
  return { status: 502, data: { error: 'TTL expired' } };
}

const forwardMessage: RelayMessage = {
  ...message,
  ttl: currentTTL - 1,
};
```

### 4. 涉及的消息类型
- `relay-request`：中继请求
- `route-query`：路由查询
- `route-response`：路由响应
- `route-update`：路由更新

## 经验总结

1. **必填 vs 可选**：TTL 字段设计为可选（`ttl?: number`），保持向后兼容，未设置时使用默认值
2. **统一处理**：所有转发逻辑都要处理 TTL，避免遗漏
3. **合理默认值**：128 跳足够覆盖绝大多数网络拓扑，同时防止无限循环
4. **错误码**：使用 502 Bad Gateway 表示 TTL 过期，语义清晰
5. **调试日志**：TTL 过期时输出调试日志，便于问题排查
