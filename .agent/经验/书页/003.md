# IndexedDB 路由表持久化经验

## 背景
用户指出 localStorage 不适合存储大规模路由表（成千上万条），建议使用 IndexedDB。

## 技术要点

### 1. IndexedDB vs localStorage
- localStorage: 5MB 限制，字符串存储
- IndexedDB: 存储容量大，支持对象存储，异步 API

### 2. TypeScript 类型处理
IndexedDB 的事件对象需要正确类型转换：
```typescript
// 错误
request.onupgradeneeded = (e) => {
  const database = e.target.result; // EventTarget 没有 result 属性
}

// 正确
request.onupgradeneeded = (e: IDBVersionChangeEvent) => {
  const req = e.target as IDBOpenDBRequest;
  const database = req.result;
}
```

### 3. 数据库设计
```typescript
数据库: peerjs-routing-db (版本 1)
├── routing-table (keyPath: target)
│   └── 路由表条目
└── direct-nodes (keyPath: nodeId)
    └── 直连节点
```

### 4. 路由表维护策略
- 每 60 秒清理超过 5 分钟未更新的路由
- 每 30 秒周期广播路由更新
- 通信失败时移除失效路由
- 容量限制：直连节点 5 个，路由表 50 个目标

## 相关文件
- src/RoutingDB.ts - IndexedDB 持久化模块
- src/Routing.ts - 路由管理器（集成持久化）
- src/PeerJsWrapper.ts - 主类（初始化和保存路由表）

## send 方法路由优先策略

### 需求变更
用户提出：优先查路由表，根据路由表确定是直连还是转发，而不是优先直连。

### 新逻辑
```
1. 查路由表 → 有路由 → 尝试中继 → 全部失败 → 降级直连 → 失败 → 结束
2. 路由表无目标 → 直连 → 失败 → 结束
```

### 设计思考
1. **路由优先**：已有路由时直接使用，避免直连尝试的延迟
2. **降级机制**：中继失败后降级到直连，作为兜底方案
3. **错误处理**：区分 HTTP 错误（直接抛出）和连接错误（尝试路由发现）
4. **路由清理**：失败时移除失效路由，保持路由表新鲜

### 代码实现要点
- 拆分错误处理逻辑到独立方法 `handleSendError()`
- 降级直连逻辑到独立方法 `fallbackToDirect()`
- 路由发现只在路由表非空时执行
