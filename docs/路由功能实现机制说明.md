# 路由功能实现机制说明

## 概述

本文档详细介绍 P2P 通信中的路由机制实现，包括路由表管理、路由发现、路由持久化等功能。

## 核心概念

### 1. 直连节点 (Direct Nodes)

直连节点指成功建立 P2P 连接的邻居节点。每个直连节点记录：
- `nodeId`: 节点 ID
- `latency`: 直连延迟（毫秒）
- `timestamp`: 最后通信时间

直连节点上限：5 个（按延迟排序，保留最优）

### 2. 路由表 (Routing Table)

路由表记录通过中继节点可达的目标节点。结构：

```typescript
interface RouteEntry {
  target: string;           // 目标节点 ID
  nextHops: NextHop[];      // 下一跳列表（按延迟升序）
  hops: number;             // 最小跳数
  timestamp: number;        // 更新时间戳
}

interface NextHop {
  nodeId: string;    // 下一跳节点 ID
  latency: number;   // 到目标的延迟（毫秒）
}
```

路由表上限：50 个目标

### 3. 中继消息类型

| 类型 | 说明 |
|------|------|
| `relay-request` | 中继请求 |
| `relay-response` | 中继响应 |
| `route-update` | 路由更新（广播可达节点） |
| `route-query` | 路由发现查询 |
| `route-response` | 路由发现响应 |

## 发送流程

当调用 `send(targetId, path, data)` 时：

```
┌─────────────────────────────────────────────────────────┐
│  1. getNextHopsToTarget(targetId)                       │
│     ↓ 有路由                                             │
│  2. tryRelayChain() → 尝试中继转发                        │
│     ↓ 失败                                               │
│  3. fallbackToDirect() → 降级直连                         │
│     ↓ 失败                                               │
│  4. handleSendError() → 移除失效路由                      │
│     ↓ 路由表非空                                          │
│  5. performRouteDiscovery() → 路由发现                   │
│     ↓ 失败                                               │
│  6. 抛出错误: "Cannot reach X"                           │
│     ↓ 无路由                                              │
│  7. 抛出错误: "Cannot reach X: no route found"          │
│                                                         │
│  8. getNextHopsToTarget(targetId) = []                  │
│     ↓ 无路由                                              │
│  9. tryDirectConnect() → 直连尝试                         │
│     ↓ 失败                                               │
│  10. handleSendError() → 同上流程                        │
└─────────────────────────────────────────────────────────┘
```

### 关键逻辑

1. **路由优先**：先查路由表，有路由则中继转发
2. **降级直连**：中继全部失败后，降级尝试直连
3. **HTTP 错误直接抛出**：404/500 等业务错误不尝试中继
4. **连接错误处理路由**：超时/连接关闭时移除失效路由，尝试路由发现
5. **路由表为空直接结束**：不再广播路由发现，减少无效网络请求
6. **失败时清理路由**：移除失效的路由条目

## 路由发现机制

### 触发条件

- 直连失败
- 路由表非空但无直达目标的路由

### 工作流程

```
┌─────────────────────────────────────────────────────────┐
│  A 需要找到 C (A -> ? -> C)                            │
│                                                         │
│  1. A 向所有直连节点广播 route-query                    │
│     消息包含: queryOrigin=A, targetNode=C               │
│                                                         │
│  2. 收到查询的节点 B:                                   │
│     - 如果自己就是 C → 返回 route-response               │
│     - 如果路由表有到 C 的路由 → 返回 route-response     │
│     - 否则 → 转发查询给其他邻居                          │
│                                                         │
│  3. A 收到响应后，将新路由加入路由表                     │
│  4. 通过新路由转发原始请求                               │
└─────────────────────────────────────────────────────────┘
```

## 路由表维护

### 1. 定时清理

每 60 秒检查一次，清理超过 5 分钟未更新的路由条目。

```typescript
setInterval(() => {
  cleanupExpiredEntries();
}, 60000);
```

### 2. 周期广播

每 30 秒向所有直连节点广播可达节点列表。

```typescript
setInterval(() => {
  broadcastRouteUpdate();
}, 30000);
```

### 3. 通信失败清理

直连或中继失败时，移除对应的失效路由：

```typescript
routingManager.removeRoute(nodeId);
```

### 4. 容量限制

- 直连节点：超出上限时移除延迟最高的
- 路由表：超出上限时按策略处理

## 路由表持久化

### 存储方式

使用 IndexedDB 存储，不受 localStorage 5MB 限制。

### 数据库结构

```javascript
数据库: peerjs-routing-db (版本 1)

对象仓库: routing-table
├── keyPath: target
└── 索引: timestamp

对象仓库: direct-nodes
├── keyPath: nodeId
└── 索引: timestamp
```

### 生命周期

1. **初始化时加载**: `routingManager.init()` 从 IndexedDB 读取
2. **运行时更新**: 路由变化时实时保存
3. **销毁时保存**: `destroy()` 调用 `persist()` 完整保存

## 类设计

### RoutingManager

```typescript
class RoutingManager {
  // 核心方法
  recordDirectNode(nodeId, latency): void      // 记录直连节点
  canReachDirectly(targetId): boolean           // 检查是否可直连
  findNextHopToTarget(targetId): NextHop | null // 查找下一跳
  getNextHopsToTarget(targetId): NextHop[]      // 获取所有下一跳
  getRoutingTable(): Record<string, RouteEntry> // 获取路由表
  
  // 路由发现
  discoverRoute(targetId): Promise<RouteEntry | null>
  handleRouteQuery(fromPeerId, message): void
  handleRouteResponse(fromPeerId, message): void
  
  // 路由更新
  broadcastRouteUpdate(): Promise<void>
  handleRouteUpdate(fromPeerId, message): void
  
  // 维护
  removeRoute(nodeId): void                     // 移除失效路由
  isRoutingTableEmpty(): boolean               // 检查路由表是否为空
  
  // 生命周期
  init(): Promise<void>                         // 初始化（加载+启动定时器）
  persist(): Promise<void>                      // 持久化到 IndexedDB
  destroy(): void                               // 销毁（停止定时器）
}
```

### IndexedDB 操作

## 调试模式

启用 `isDebug=true` 时，会输出路由相关日志：

```
Routing directNode { nodeId, latency }
Routing update { target, nextHop, latency }
Routing discovered { targetNode, nextHop, latency }
Routing cleanup { routes, nodes }
Routing routeRemoved nodeId
```

## 常见错误

| 错误信息 | 说明 |
|---------|------|
| `Cannot reach X: no route found` | 直连失败且路由表为空 |
| `Cannot reach X: no route found and routing table is empty` | 路由表为空，无法中继 |
| `All relay nodes failed` | 所有中继节点都失败 |
| `timeout` | 连接超时（30秒） |
